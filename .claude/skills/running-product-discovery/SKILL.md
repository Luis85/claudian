# Running Product Discovery

## Overview

Discovery is **risk-reduction, not documentation**. Its job is to answer four questions *before* committing build capacity, and to replace opinion (the loudest voice in the room) with evidence from real customers.

This skill is the entry point and router for a discovery effort. It orients the cross-functional team and points to the workflow skill for each step. **It does not do the work itself** — it hands off.

**When a request arrives as a pre-formed solution ("add a referral feature", "build a dashboard"), do NOT scope, design, or estimate it.** Start at step 1 (`framing-the-opportunity`) and treat the named feature as one untested hypothesis, not the unit of work.

## The four big product risks (the running checklist)

Ask these continuously; address the *biggest* first. Teams over-index on feasibility — push on value and viability.

| Risk | Question | Owner |
|------|----------|-------|
| **Value** | Will customers buy it / users choose it? | Product Manager |
| **Usability** | Can users figure out how to use it? | UX Designer |
| **Feasibility** | Can engineers build it (time/skills/tech)? | Engineer |
| **Business viability** | Does it work for legal, finance, sales, brand? | PM + stakeholders |

## Who does what

Discovery is a **trio** (PM + designer + engineer) that decides *and* builds together — not a relay of hand-offs. A requirements engineer / BA is a valuable fourth seat for data-heavy, regulated, or enterprise work, owning rigor, traceability, and testability. Keep customer contact shared: interview together, don't filter reality for each other.

## The discovery arc → which skill to use

Work iteratively, not strictly top-to-bottom.

| Step | Use this skill |
|------|----------------|
| 1. Frame the problem (outcome, HMW, POV) | `framing-the-opportunity` |
| 2. Surface & rank assumptions, design tests | `mapping-discovery-assumptions` |
| 3. Plan & run customer research | `planning-discovery-interviews` |
| 4. Synthesize notes into evidence-linked insight | `synthesizing-discovery-research` |
| 5. Model the user | `building-evidence-based-personas` |
| 6. Model the need / motivation | `defining-jobs-to-be-done` |
| 7. Model the experience | `mapping-customer-journeys` |
| 8. Connect work to measurable outcomes | `mapping-impact-to-outcomes` |
| 9. Shape the solution into releases | `story-mapping-the-solution` |
| 10. Specify stories, acceptance criteria, NFRs | `writing-requirements` |
| 11. Prioritize | `prioritizing-with-evidence` |

## AI guardrails (non-negotiable in every research step)

AI accelerates *drafting, structuring, and critiquing* discovery artifacts. It must never become a fabrication engine. Five rules:

1. **Treat every AI output as a hypothesis to be tested, never as a finding.**
2. **Keep real customer contact** — AI complements research with real users; it never replaces it.
3. **Trace every claim to evidence** — every insight links to a real transcript/observation; never invent quotes, participants, or "synthetic users."
4. **Seek disconfirming evidence** — ask "what would prove this wrong?", not just "what supports it?"
5. **Protect participant data** — no raw PII into prompts without consent and a defined scope.

## Anti-patterns this suite prevents

Feature factory (output over outcome) · solution-jumping (building before understanding) · HiPPO decisions · confirmation bias · leading questions · analysis paralysis · ambiguous/untestable requirements · gold-plating / scope creep.
