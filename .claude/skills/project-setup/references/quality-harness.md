# Quality harness reference

## Guardrails

Each guardrail is a toggle in `answers.json` under `guardrails`. All default to `true` (enabled). A disabled guardrail is never installed — its script, config, and CI step are omitted entirely.

### `eslintSeverityStaging`
Installs ESLint with a flat config (`eslint.config.mjs`). Rules use two tiers: errors block CI immediately; warnings form a tracked backlog. Promote a rule from `warn` to `error` once all its instances reach zero. This staging model lets brownfield projects adopt linting incrementally without blocking CI on day one.

### `locGuard`
Installs a per-file LOC ratchet (`scripts/check-loc.mjs`). Files exceeding `locCap` (default 500 lines) fail CI. The baseline snapshots the current maximum so no existing file is immediately red. New files must stay under the cap.

### `fallowRatchet`
Installs the fallow metric ratchet (`scripts/check-quality.mjs`). fallow measures dead code, duplication, and complexity (CRAP score). The baseline records today's counts; CI fails if any metric regresses. **Critical caveat:** always run `check:quality` with `./coverage` absent. A stale coverage artifact causes fallow to compute dynamic CRAP instead of static-estimated CRAP, inflating critical-complexity counts and producing false positives.

### `coverageFloors`
Installs coverage thresholds in the test framework config. The baseline runs `test:coverage`, measures the current summary, and floors each metric (statements/branches/functions/lines) at `Math.floor(pct)` — rise-only. If the test config is user-authored (no "Generated by project-setup" marker), the engine warns rather than rewriting it, and the floor is not set.

### `ci`
Requires `github.integrate: true`. Writes `.github/workflows/ci.yml` with steps for each enabled guardrail. Uses the detected package manager (npm/pnpm/yarn).

## Ratchet + severity-staging mechanics

- **Fallow ratchet:** Snapshots are stored in `scripts/quality-baseline.json`. Each `check:quality` run compares current counts against the snapshot. Any regression (increase) fails. To tighten the ratchet, run `node scripts/check-quality.mjs --update` after improving the code.
- **LOC ratchet:** Snapshot is the current maximum LOC across all files. Stored as a single number in `scripts/loc-baseline.json`. Each `check:loc` run rejects any file that now exceeds both the cap and the snapshot.
- **Coverage floor:** Stored as thresholds in the test framework config itself. Jest enforces them on every `test:coverage` run.

## `answers.json` shape (options object)

```json
{
  "testFramework": "jest",
  "locCap": 500,
  "packageManager": "npm",
  "guardrails": {
    "eslintSeverityStaging": true,
    "locGuard": true,
    "fallowRatchet": true,
    "coverageFloors": true,
    "ci": true
  },
  "docs": {
    "scaffold": true
  },
  "github": {
    "integrate": false,
    "mcp": false
  }
}
```

`testFramework` defaults to the detected framework (jest or vitest). `packageManager` defaults to the detected one. All guardrails default to `true` when omitted.
