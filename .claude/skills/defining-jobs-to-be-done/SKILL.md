# Defining Jobs-to-be-Done

## Overview

JTBD focuses on the **progress a customer is trying to make in a circumstance** — what they "hire" a product to do — not their demographics. It has functional, emotional, and social dimensions (the milkshake is hired to make a boring commute better, not because of its flavor).

JTBD is **three different schools**. Pick the one that fits the question — don't blend the jargon.

## Pick the right school

| School | Use when you need to… | Signature tool |
|--------|----------------------|----------------|
| **Christensen** (Jobs Theory) | Reframe the problem space — what progress, in what circumstance, competing against what | The struggling-moment narrative (milkshake) |
| **Klement / Moesta** (Forces) | Understand *why customers switch or don't* | Switch interview + forces of progress |
| **Ulwick** (ODI) | Produce a *quantified, prioritized* list of unmet needs | Job map + desired-outcome statements |

JTBD and personas are complementary: JTBD = *what progress*; personas = *who, and which trade-offs to prioritize*.

## Tools

**Job story** (situation-anchored): **"When [situation], I want to [motivation], so I can [expected outcome]."** The "When" is load-bearing — it ties the job to a real circumstance.

**Switch interview + four forces of progress** — reconstruct the timeline of a real switch (first thought → passive looking → active looking → deciding → first use → ongoing use) and map the four forces:
- **Push** of the situation (dissatisfaction with the status quo) — drives change
- **Pull** of the new solution (its appeal) — drives change
- **Anxiety** about the new (will it work?) — blocks change
- **Habit** / allegiance to the present (inertia) — blocks change

A switch happens only when **Push + Pull > Anxiety + Habit**. Listen for emotional energy; probe vague language into concrete comparisons.

**Job map (Ulwick, 8 universal steps)** — deconstruct the functional job, solution-free: **Define → Locate → Prepare → Confirm → Execute → Monitor → Modify → Conclude.** Build the map first, then capture **desired-outcome statements** per step (*direction + metric + object + context*, e.g., "minimize the time it takes to…"), which can later be scored for importance vs. satisfaction.

## Workflow

1. Pick the school for your question.
2. For switch work: recruit recent switchers, reconstruct the timeline, map the forces.
3. For ODI: define the core functional job, build the job map, write measurable outcome statements.
4. Output job statements / forces diagram / job map → feed `framing-the-opportunity`, `mapping-customer-journeys`, or `prioritizing-with-evidence`.

## Quality bar

- [ ] Job statements are solution-free (describe progress, not features).
- [ ] Switch findings come from real recent switchers, not speculation.
- [ ] Desired outcomes are measurable (direction + metric + object + context).
- [ ] Job/outcome statements drawn from real user language are marked as evidence; invented ones are explicitly labeled hypotheses to validate.
